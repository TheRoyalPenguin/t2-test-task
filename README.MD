## Ситуация 3: Тормозит SQL запрос
### Диагностика:
1) Выполню данный запрос напрямую в СУБД с добавлением в начало `EXPLAIN ANALYZE` (для `PostgreSQL`). Это покажет мне план выполнения запроса: используются ли индексы (`index scan`) или же таблица сканируется целиком (`seq scan`). 
2) Дополнительно посмотрю текущие блокировки (таблицы `pg_stat_activity` и `pg_locks`), чтобы исключить ожидание освобождения ресурса другой транзакцией.
Варианты исправления:
1) При отсутствии индекса: если таблица большая, создам составной индекс для ускорения поиска: `CREATE INDEX idx_users_id_active ON users (user_id, is_active);`
2) Если колонка `user_id` в таблице имеет тип `VARCHAR`, то с текущим запросом СУБД конвертирует каждый `user_id` на лету, что делает использование индекса невозможным. Поэтому исправлю в запросе на `user_id = '123'`.
3) Обновлю статистику БД, запустив `VACUUM ANALYZE users;`, чтобы планировщик запросов построил оптимальный маршрут.
4) Если проблема в блокировке, предложу разработчикам оптимизировать длительность транзакций в их коде.

## Ситуация 4: Ошибка 403 forbidden в Docker
### Причина: 
Служба `Docker` на сервере настроена на работу через корпоративный прокси-сервер (переменные `HTTP_PROXY`/`HTTPS_PROXY`). Когда `Docker` пытается скачать образ с приватного локального репозитория `foo.bar.ru`, запрос уходит на прокси, который блокирует доступ (выдает `403 Forbidden`), так как прокси не маршрутизирует трафик во внутреннюю сеть или требует явной авторизации.
### Решение: 
Добавлю домен `foo.bar.ru` в исключения для прокси с помощью переменной `NO_PROXY` в настройках `systemd` службы `Docker`:
1) `sudo systemctl edit docker`
2) Добавлю настройки в открывшийся файл:
```
`[Service]`
`Environment="HTTP_PROXY=http://proxy.domain.com:80"`
`Environment="HTTPS_PROXY=http://proxy.domain.com:80"`
`Environment="NO_PROXY=localhost,127.0.0.1,foo.bar.ru"`
```
3) `sudo systemctl daemon-reload`
   `sudo systemctl restart docker`

## Ситуация 5: Ansible роль

### Описание решения:
`Ansible` проект разделен на роли и теги для возможности выборочного запуска. Используется шаблонизация `Jinja2` для подстановки конфигурации.

### Как запустить проект:
1. Полная установка и запуск: `ansible-playbook -i inventory.ini playbook.yml`
2. Запуск этапа подготовки (установка `Docker` и каталогов): `ansible-playbook -i inventory.ini playbook.yml --tags "prepare"`
3. Запуск только этапа конфигурации (шаблонизация `config` файлов): `ansible-playbook -i inventory.ini playbook.yml --tags "config"`

Данные для подключения к серверам вынесены в `inventory.ini`, а переменные окружения в `group_vars/grafana.yml`. Владельцем создаваемых директорий указан `UID 472`, так как офф `Docker` образ `Grafana` работает от этого пользователя. Также добавлен `healthcheck` для проверки статуса `200 OK` от `API Grafana` после старта.
